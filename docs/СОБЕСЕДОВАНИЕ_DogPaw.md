Документ по проекту `dog-owner-app`: можно читать как **введение для людей не из IT**, и как **технический конспект** для интервью. Термины с пояснениями; в конце — короткие формулировки «как сказать вслух».

## 1. Что такое DogPaw простыми словами

**DogPaw** (в репозитории часто фигурирует как приложение-помощник владельца собаки) — это **мобильное приложение на телефон** (iOS и Android через одну кодовую базу), плюс **отдельная программа-сервер в интернете** (бэкенд), которая хранит данные пользователей и отдаёт их приложению по запросу.

**Задача продукта:** собрать в одном месте то, что собаковод делает разрозненно — карточку питомца, прогулки, карту полезных мест, общение с другими владельцами, статьи об уходе и экстренные объявления, если питомец потерялся.

**Как это работает в быту:** пользователь ставит приложение, регистрируется, заводит карточку собаки. Дальше приложение может записывать прогулки (с телефона берётся геолокация), показывать на карте клиники и парки, показывать ленту постов других пользователей и т.д. Всё, что должно **сохраняться между переустановками и с других устройств**, уходит на сервер в **базу данных**.

## 2. Из чего состоит система (три слоя)

Представьте **официанта в ресторане**:

1. **Клиент (приложение на телефоне)** — это гость: смотрит меню, делает заказ, получает блюдо. В нашем случае экраны React Native: «Питомец», «Прогулки», «Карта» и т.д.
2. **Сервер (бэкенд на Go)** — это кухня и официант вместе: принимает заказ по понятным правилам (HTTP-запросы), готовит ответ, иногда ходит к **другим** поставщикам (картам Яндекса или OpenStreetMap) за данными о местах.
3. **База данных (SQLite)** — это склад ингредиентов: пользователи, питомцы, прогулки, посты. Пока это один файл на диске сервера — удобно для разработки и небольшого деплоя.

**Связь между слоями:** приложение не подключается к базе напрямую. Оно говорит с сервером на языке **HTTP** (как «заказ по форме»), обычно тело запроса и ответа — **JSON** (текстовый формат «поле — значение», удобный и для человека, и для программ).

## 3. Глоссарий (термины, которые могут спросить или которые стоит произнести правильно)

| Термин | Простыми словами |
|--------|------------------|
| **Мобильное приложение** | Программа на телефоне; здесь одна кодовая база на **React Native** + **Expo** (инструменты сборки и запуска без нативной студии для каждой правки в простом сценарии). |
| **TypeScript** | JavaScript с проверкой типов на этапе разработки — меньше глупых ошибок в интерфейсе. |
| **REST API** | Набор **адресов URL** и правил: «GET — взять данные», «POST — создать», «PUT — обновить», «DELETE — удалить». Наш API начинается с префикса `/api/v1` (версия в пути). |
| **JSON** | Формат обмена данными: объекты `{ "ключ": "значение" }`, массивы. И клиент, и сервер его понимают. |
| **Go (Golang)** | Язык серверной части: быстрая компиляция, один исполняемый файл, хорошо читается. |
| **Gin** | Библиотека для Go: маршрутизация HTTP («этот URL → эта функция»), middleware. |
| **GORM** | **ORM** — «переводчик» между таблицами в БД и структурами в коде: меньше ручного SQL для типовых операций. |
| **SQLite** | Встраиваемая база: один файл, без отдельного процесса СУБД. Для учебного/личного проекта — просто; для большой нагрузки обычно переходят на PostgreSQL и т.п. |
| **JWT (JSON Web Token)** | Подписанный «билет» после логина: сервер проверяет подпись и достаёт из билета id пользователя, не храня сессию в таблице сессий (упрощённо — **stateless**). |
| **bcrypt** | Медленная хэш-функция для паролей: в базе лежит не сам пароль, а хэш; при вводе пароля сравнивается хэш. |
| **Bearer-токен** | Способ передать JWT в заголовке: `Authorization: Bearer <токен>`. |
| **Middleware** | «Прослойка» на сервере: до вызова обработчика проверить токен, залогировать, ограничить частоту запросов. |
| **CORS** | Правила для **браузера**, с каких сайтов можно звать API. У нас выставлено широко (`*`) — для нативного приложения не критично; для веб-сайта в проде обычно ужесточают. |
| **Rate limit** | Ограничение числа запросов с одного IP за время — защита от случайного злоупотребления и простых атак. |
| **Overpass / OSM** | **OpenStreetMap** — карта мира сообществом. **Overpass API** — язык запросов «дай объекты в радиусе». Не требует API-ключа, но зависит от доступности публичных зеркал. |
| **EAS (Expo Application Services)** | Облачная сборка и публикация приложений в App Store / Google Play по конфигу проекта. |

| Слой | Технологии | Зачем так (для объяснения) |
|------|------------|----------------------------|

## 5. Как устроено приложение изнутри (поток экранов)

1. Запуск → **UserProvider** читает из AsyncStorage сохранённый токен.
2. Если токен есть → запрос **`GET /auth/me`** на сервер: актуальный профиль; при ошибке сессия сбрасывается.
3. Если пользователь **не** залогинен → экран **AuthScreen** (регистрация / вход).
4. Если залогинен → **AppNavigator**: нижние вкладки с основными разделами.

**Сессия:** в памяти клиента держится функция `setAuthToken` — все последующие `fetch` подставляют заголовок `Authorization`. При выходе токен очищается и удаляется из AsyncStorage.

## 6. Сервер: маршруты и смысл разделов

Все бизнес-эндпоинты под **`/api/v1`** — это **версионирование API**: если когда-нибудь сломаем совместимость, можно ввести `v2`, не ломая старых клиентов сразу.

**Краткая карта (не все поля):**

- **Auth:** регистрация, логин, «кто я», выдача админки другу (ограниченный сценарий).
- **Пользователи и питомцы:** профиль, создание/обновление питомца, **поиск питомца по QR-коду** (публичный сценарий).
- **Прогулки:** список по питомцу, создание записи, удаление (с проверкой прав).
- **Места:** список из БД, **поиск в интернете** по координатам, отзывы.
- **Карта друзей:** позиции пользователей, обновление своей.
- **Друзья:** список, добавить, удалить.
- **Лента:** посты, лайки, комментарии.
- **Статьи:** список/детали, подача статьи пользователем, модерация админом.
- **Потеряшки:** список объявлений, создать, отметить «найден».

Дополнительно: **`GET /health`** — проверка «сервер жив» (удобно для мониторинга и ручной проверки).

**Защита:** часть маршрутов без токена (например публичная лента, статьи, QR питомца), часть **только с JWT** (друзья, удаление, модерация). Для статей иногда используется **опциональный** JWT: гость видит одно, залогиненный — с учётом прав.

## 7. Функции продукта — подробно (для неспециалиста + техника)

### 7.1. Питомец и QR-код

**По-человечески:** у каждой собаки в приложении есть карточка. У карточки есть **уникальный код** (в том числе для QR на ошейнике). Если кто-то отсканирует QR, приложение или браузер может запросить **публичный** эндпоинт и показать информацию для контакта владельца — в зависимости от того, что вы решите отдавать в API.

**Техника:** в модели питомца поле `qr_code_data`; при создании, если пусто, подставляется `pet-{id}`. Эндпоинт вида **`GET /api/v1/pets/qr/:qr`** возвращает данные питомца по этому коду.

---

### 7.2. Трекер прогулок

**По-человечески:** пользователь нажимает «начать прогулку», телефон периодически спрашивает GPS «где я», приложение считает **пройденный путь**, время и приблизительно **калории**, по окончании отправляет запись на сервер.

**Техника (важно для рассказа):**

- Используется **`expo-location`** (подписка на обновления координат).
- Расстояние между точками — **формула гаверсинуса** (расстояние по сфере Земли в километрах).
- Чтобы отсечь **скачки GPS** и шум: не суммируются слишком длинные сегменты (порог в коде ~0.25 км), не копятся слишком короткие (шум), отбрасываются точки с грубой точностью (если поле accuracy доступно).
- Маршрут хранится как **JSON-массив** точек `{ lat, lng }` в поле прогулки в БД (как текст/строка — для SQLite это нормально на масштабе проекта).
- На сервер уходит создание записи **`POST /walks`** с расстоянием, длительностью, калориями, маршрутом и временем.

Это хороший пример **обработки ненадёжных реальных данных** (GPS) — такие темы на собеседованиях любят.

---

### 7.3. Калькулятор кормления

**По-человечески:** экран, где вводят вес собаки, возраст, уровень активности — приложение **на телефоне** считает рекомендуемые граммы корма в день. Переключатель «напоминания» в UI заложен как идея; логика напоминаний может быть не подключена к пушам (уточняй по факту кода — в выжимке это **клиентский калькулятор**).

**Техника:** формула в `FeedingCalculatorScreen`: базовые граммы на кг зависят от возрастных порогов (&lt;1 года, &lt;2 лет, старше), множитель по активности (низкая / средняя / высокая). **Бэкенд для этой формулы не обязателен** — нет необходимости тащить на сервер то, что не нужно хранить.

---

### 7.4. Карта мест (ветеринар, парк, магазин…)

**По-человечески:** пользователь смотрит карту в своей точке; приложение просит сервер: «найди в радиусе N километров точки интереса по категориям».

**Техника:**

- Эндпоинт **`GET /places/search`** с параметрами `lat`, `lng`, радиус, лимит, список категорий, флаг `save` (сохранять ли найденное в локальную БД приложения на сервере).
- **Порядок источников в коде:** если задан **`YANDEX_MAPS_API_KEY`**, сначала запрашивается **Яндекс**; если результатов нет — **OpenStreetMap через Overpass**. Если ключа нет — сразу OSM.
- Для OSM в коде перечислены **несколько зеркал** Overpass и **повторы** при ошибке — это устойчивость к падению одного узла.
- Результаты **кэшируются в памяти сервера** на несколько минут по ключу области (чтобы не ддосить внешние API при движении карты туда-сюда). При переполнении кэша — грубый сброс (эвристика).
- Опционально найденные места **сохраняются в SQLite** с дедупликацией по имени и адресу (обновление координат/категории).

**Важно для честности:** в `backend/README.md` описание порядка источников может отличаться от кода — на собеседовании лучше опираться на **фактическую логику в `handlers/places.go`**.

---

### 7.5. «Умная карта» (где друзья)

**По-человечески:** пользователь может делиться приблизительным местоположением и статусом («ищу компанию», «тренируюсь», «не беспокоить»), видеть других на карте.

**Техника:** сущность **UserLocation**, обновление **`PUT /map/me`**, список **`GET /map/users`** (под авторизацией). Статусы перечислены в модели (`WalkStatus`).

---

### 7.6. Друзья

**По-человечески:** добавление в друзья по id или email, список друзей, удаление. Нужен вход в аккаунт.

**Техника:** таблица **Friendship**, эндпоинты под JWT.

---

### 7.7. Лента

**По-человечески:** лента постов с текстом и опционально картинкой, лайки и комментарии.

**Техника:** посты, отдельные сущности лайков и комментариев; при отдаче ленты подтягиваются имена авторов. Сортировка через query **`sort`**: по умолчанию **interesting** — эвристика «интересности» (например вес лайков и комментариев), также **newest**, **likes**, **comments**.

---

### 7.8. Энциклопедия (статьи)

**По-человечески:** статьи по категориям; пользователь может **предложить** статью; **модератор** публикует или отклоняет.

**Техника:** поля статуса `pending` / `published` / `rejected`, эндпоинты подачи и модерации. Доступ к модерации у пользователей с флагом администратора.

---

### 7.9. Потеряшки (Lost & Found)

**По-человечески:** объявление «питомец потерян» с контактом и точкой на карте; можно отметить «найден».

**Техника:** сущность **LostPetAlert**, фильтр по статусу в списке.

## 8. Безопасность и ограничения (честно и понятно)

**Что сделано разумно для учебного/личного проекта:**

- Пароли **не хранятся в открытом виде** (bcrypt).
- Доступ к личным действиям по **JWT**.
- Простой **rate limit** на IP (120 запросов в минуту) — сглаживает случайные штормы и простейший спам.
- Клиент с **таймаутом** и **одной повторной попыткой** на сетевые сбои.

**О чём говорить как о дорожной карте, а не как о провале:**

- **JWT без refresh и blacklist** — отозвать токен до истечения срока без дополнительной инфраструктуры нельзя; для продакшена часто делают короткий access + refresh, либо серверные сессии.
- **Rate limit в памяти одного процесса** — при нескольких инстансах за балансировщиком лимит «плавает»; нужен общий store (Redis и т.д.).
- **SQLite** — один файл, один writer по сути; при росте нагрузки и команд — **PostgreSQL** и миграции.
- **Секрет JWT** в коде есть **заглушка для разработки**; в бою обязательно **переменная окружения `JWT_SECRET`**.
- **CORS `*`** — для нативного клиента не боль; для веба — сузить.
- **Админ-права:** в коде заложены сценарии выдачи прав владельцем только **друзьям** — это осознанное ограничение «круг доверия», не «все подряд админы».

## 9. Нефункциональные детали клиента (`client.ts`)

Единый слой запросов:

- Базовый URL из конфига (для телефона в той же Wi‑сети часто указывают IP компьютера — иначе `localhost` на телефоне укажет на сам телефон).
- **Таймаут** по умолчанию 10 секунд; для тяжёлого поиска мест — увеличенный (десятки секунд), потому что внешние API могут отвечать долго.
- **Повтор** один раз при ошибке — полезно при мигании сети.
- **`AbortSignal`** — если пользователь быстро сдвинул карту, старый запрос можно отменить, не смешивая ответы.

Это хорошая аналогия с **внутренними HTTP-клиентами** в больших компаниях: политики таймаутов и ретраев как часть платформы.

## 10. Ответы на типовые вопросы (развёрнуто)

**В чём твоя роль?**  
Я прошёл путь от модели данных и API до экранов приложения: спроектировал сущности, реализовал REST, интеграцию внешних карт, клиентский слой запросов, ключевые сценарии UI и навигацию, сиды для демо, конфигурацию сборки через EAS.

**Почему Go?**  
Предсказуемый серверный код, один бинарник для деплоя, Gin даёт быстрый REST, GORM ускоряет работу с моделями без ручного SQL на каждый чих.

**Почему React Native / Expo?**  
Одна кодовая база на две платформы; Expo снижает порог входа и ускоряет итерации на реальном устройстве.

**Как работает логин?**  
Регистрация хэширует пароль и создаёт пользователя; логин проверяет пароль и выдаёт JWT; клиент сохраняет токен и подставляет в заголовки; сервер в middleware вытаскивает `user_id` из токена.

**Как бы масштабировал?**  
PostgreSQL, вынести медиа в объектное хранилище, горизонтально масштабировать API, Redis для кэша и rate limit, очереди для тяжёлых задач, отдельный поисковый контур для мест при нагрузке, мониторинг и алерты.

**Где самое интересное в коде?**  
Прогулки: фильтрация шума GPS и скачков перед суммированием дистанции. Или поиск мест: два провайдера, fallback, кэш и устойчивость к отказу зеркал OSM.

**Чего не хватает из инженерной дисциплины?**  
Автотесты бэкенда (`httptest`), контрактные тесты API, E2E на клиенте — честно назвать и сказать, что это следующий шаг.

## 11. Мини-скрипты «как сказать вслух»

### 30 секунд (совсем широкая аудитория)

«Я сделал приложение для владельцев собак: карточка питомца с QR, прогулки с маршрутом, карта полезных мест из интернета, друзья и лента, статьи и объявления о пропаже. Клиент на телефоне общается с сервером на Go, данные в базе; можно собрать и опубликовать в магазинах через Expo.»

### 2–3 минуты (интервью)

1. Проблема и аудитория.  
2. Два компонента: мобильный клиент (Expo, TypeScript) и сервер (Go, REST, SQLite).  
3. Три опорных блока: **питомец + прогулки с GPS**, **карта и поиск мест с внешними API и кэшем**, **социальное + контент + lost&found**.  
4. Одно техническое углубление на выбор: GPS/маршрут или два источника карт + fallback.  
5. Ограничения и как бы рос: PostgreSQL, нормальный rate limit, тесты.
