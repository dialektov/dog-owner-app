## Что сделано

### Реальные данные и хранение по пользователям

- **Backend:** API уже хранит данные в SQLite (файл `backend/data/dogowner.db`). Добавлен возврат имени пользователя в ответе `GET /map/users` для отображения на умной карте.
- **Клиент:** Введён контекст текущего пользователя (`UserContext`): при первом запуске создаётся гостевой пользователь через `POST /users`, его `id` и данные сохраняются в AsyncStorage. Все последующие запросы (карта, питомец и т.д.) привязаны к этому пользователю. Данные каждого пользователя хранятся на сервере в таблицах `users`, `pets`, `walks`, `user_locations` и др.
- **Экран питомца:** Загружает пользователя по `id` из контекста, затем первого питомца с API (`GET /users/:id`, `GET /pets/:id`). Если питомца нет — показывается сообщение (для демо можно выполнить seed на сервере).

### Динамическая умная карта с геопозицией

- **Карта мест:** Маркеры мест загружаются с API (`GET /places`) вместо захардкоженного списка.
- **Умная карта:**
  - Список пользователей на карте приходит с API (`GET /map/users`) с полями `user_name`, `latitude`, `longitude`, `status`.
  - Текущий пользователь отправляет свою геопозицию на сервер (`PUT /map/me`) при включённом режиме «Умная карта»: запрашивается разрешение на геолокацию (expo-location), позиция обновляется по таймеру (каждые ~20 с) и при изменении положения.
  - Список пользователей на карте обновляется каждые 15 с (polling).
  - Добавлен выбор своего статуса: «Ищем компанию», «На тренировке», «Не беспокоить» — значение уходит в `PUT /map/me` и отображается у маркера на карте у других пользователей.

### Регистрация пользователей

- **Экран «Профиль»:** Добавлена вкладка «Профиль» в нижней панели. На экране отображаются текущий пользователь (имя, email) и, если пользователь гость (email вида `...@dogpaw.local`), форма регистрации: поля «Email» и «Имя», кнопка «Зарегистрироваться».
- **Логика регистрации:** При нажатии вызывается `POST /api/v1/users` с введёнными email и именем. Ответ (id, name, email) сохраняется в AsyncStorage и подставляется в контекст как новый текущий пользователь. Повторная регистрация с уже занятым email возвращает ошибку от сервера — показывается сообщение «Попробуйте другой email».
- **Контекст:** В `UserContext` добавлены метод `registerUser(email, name)` и флаг `isGuest` (true, если email заканчивается на `@dogpaw.local`). После успешной регистрации гость заменяется зарегистрированным пользователем без перезапуска приложения.

### Авторизация и хранение аккаунтов (обновление)

- Внедрена полноценная auth-схема: `POST /api/v1/auth/register`, `POST /api/v1/auth/login`, `GET /api/v1/auth/me`.
- В модели пользователя добавлено хранение хеша пароля (`password_hash`), пароли хешируются через bcrypt.
- Добавлены JWT-токены (подписанные HMAC): клиент хранит токен и профиль в AsyncStorage (`@dogpaw_session`).
- Убран режим автосоздания гостя: теперь при первом запуске открывается экран входа/регистрации (`AuthScreen`), без демо-аккаунтов.
- Защищены приватные эндпоинты middleware-авторизацией (`/map/me`, `POST /friends`, `POST /feed`, `POST /lost`, `PUT /lost/:id/found`, `/auth/me`).
- Геопозиция на умной карте привязана к авторизованному пользователю по токену (user_id берётся из JWT, не из тела запроса).

### Документация

- Добавлен этот файл `docs/ОТЧЕТ_РАЗРАБОТКА.md` для фиксации проделанной работы, ошибок и решений. Документ обновляется при добавлении новых функций (в т.ч. регистрация).

### Публикация в App Store и Google Play

- Добавлена конфигурация **EAS** (`eas.json`): профили сборки `development`, `preview`, `production` и профиль отправки в магазины `submit.production` (с плейсхолдерами для Apple ID и сервисного аккаунта Google).
- Добавлена инструкция **docs/ПУБЛИКАЦИЯ_В_МАГАЗИНЫ.md**: что нужно (аккаунты Apple Developer и Google Play Console), установка EAS CLI, команды `eas build` и `eas submit`, настройка и ручная загрузка билда. Так приложение можно сделать доступным для загрузки напрямую из магазинов.

### Интеграция Яндекс Карт для dog-friendly мест

- **Backend:** Добавлен сервис `backend/services/yandex_places.go` для поиска мест через Yandex Search API (`https://search-maps.yandex.ru/v1/`) по центру карты и радиусу. Эндпоинт `GET /api/v1/places/search` принимает `lat`, `lng`, `radius_km`, `limit`, `save` и возвращает найденные места.
- **Автосохранение в БД:** При `save=true` найденные места сохраняются в SQLite, дубликаты схлопываются по паре `name + address`.
- **Фильтрация dog-friendly:** Используется набор целевых запросов (ветклиники, зоомагазины, площадки/парки для собак, dog-friendly кафе, груминг). Категории приводятся к внутренним (`vet`, `pet_shop`, `groomer`, `park`, `cafe`).
- **Клиент:** В `MapScreen` добавлена кнопка «Обновить dog-friendly из Яндекс Карт». Кнопка вызывает новый API, после чего перезагружает список `GET /places` и отображает результаты на карте.
- **Актуализация логики карты:** режим «Места» переключён на прямую загрузку отфильтрованных POI из Яндекс Карт по текущему региону карты (ветклиники, зоомагазины, груминг, парки/площадки, dog-friendly кафе). Теперь на карте показываются только подходящие категории, без ручного набора точек.

### Режим «Потерялся питомец» (Lost & Found)

- **Backend:** Добавлена сущность `LostPetAlert` и API:
  - `GET /api/v1/lost?status=active|found|all`
  - `POST /api/v1/lost`
  - `PUT /api/v1/lost/:id/found`
- **Клиент (карта):** В `MapScreen` добавлен отдельный режим «Потерялся». На карте отображаются активные объявления красными маркерами, список обновляется по таймеру.
- **Клиент (карточка питомца):** Добавлена кнопка «🚨 Потерялся питомец» на экране питомца. При нажатии берётся текущая геопозиция (если доступна) и создаётся объявление через API.
- **Seed:** В демонстрационные данные добавлен пример активного объявления о пропаже.

### Нефункциональные улучшения (реализовано)

- **Надёжность клиента:** в `api/client.ts` добавлен таймаут запросов (10 сек) и 1 повторная попытка при временной ошибке сети.
- **Стабильность backend:** добавлен rate limiter middleware (120 запросов/мин на IP), чтобы защитить API от перегрузки.
- **UX карты:** цветовая дифференциация категорий мест (ветклиники, зоомагазины, груминг, парки, dog-friendly кафе) + легенда.

### Убраны заглушки на ключевых экранах

- **Экран «Прогулки»:** вместо демо-истории подключены реальные данные из API (`GET /pets/:id/walks`), добавлен реальный старт/стоп прогулки с геолокацией, таймером и сохранением (`POST /walks`).
- **Экран «Энциклопедия»:** вместо захардкоженного списка подключена загрузка статей из API (`GET /articles`), добавлены фильтр по категориям, поиск по тексту и разворачивание полного содержания статьи.
- **Экран «Друзья и общение»:** вместо демо-данных подключены реальные друзья и лента (`GET /users/:id/friends`, `GET /feed`), добавлено добавление друга по QR-данным (`GET /pets/qr/:qr` + `POST /friends`) и публикация поста (`POST /feed`).
- **Карточка питомца:** кнопка редактирования больше не «мертвая» — показывает контролируемое сообщение о следующем шаге разработки; кнопка «Потерялся питомец» создаёт реальное объявление.

### Доработка кнопок без действий (финал)

- **Карточка питомца:** реализовано полноценное создание/редактирование карточки прямо в UI (форма + сохранение в API через `POST /pets` и `PUT /pets/:id`). Кнопка «Редактировать карточку» теперь открывает рабочую форму и сохраняет изменения.
- **Друзья:** кнопка связи `💬` теперь выполняет реальное действие — открывает email-клиент (`mailto:`) на адрес друга; если клиент недоступен, показывает контакт в Alert.
- **Итог:** на основных пользовательских экранах убраны «мертвые» кнопки, каждое нажатие приводит к явному полезному действию.

## Ошибки и решения

### 1. Несовместимость версий Expo Go и проекта (SDK 52 vs 54)

**Проблема:** На устройстве установлен Expo Go для SDK 54, проект был на SDK 52 — приложение не открывалось.

**Решение:** Обновлён проект до Expo SDK 54: в `package.json` выставлены `expo@~54.0.0`, `react@19.1.0`, `react-native@0.81.0`, `@types/react@^19.1.0`. Удалён `package-lock.json` и выполнена чистая установка (`npm install`). После этого проект открывается в актуальном Expo Go.

### 2. Конфликт peer-зависимостей при `npm install`

**Проблема:** Ошибка ERESOLVE: `react-native@0.76.x` тянул `react@^18.2.0` и `@types/react@^18.2.6`, в то время как в проекте были `react@19.1.0` и `@types/react@~19.0.0`.

**Решение:** В `package.json` зафиксированы версии под SDK 54: `react-native@0.81.0`, `@types/react@^19.1.0`. Удалён старый `package-lock.json`, чтобы npm заново разрешил зависимости под новые версии.

### 3. Отсутствует модуль `babel-preset-expo`

**Проблема:** При запуске приложения ошибка: `Cannot find module 'babel-preset-expo'`.

**Решение:** В `devDependencies` добавлен пакет `babel-preset-expo` (версия ~13.0.0 для SDK 54). Выполнен `npm install`.

### 4. На умной карте не отображались имена пользователей

**Проблема:** Эндпоинт `GET /map/users` возвращал только `user_id`, без имени — на карте было непонятно, кто есть кто.

**Решение:** В бэкенде введён тип `UserLocationResponse` с полем `user_name`. В обработчике `GetUserLocations` для каждой записи выполняется выборка имени из таблицы `users` по `user_id` и заполнение `UserName` в ответе.

### 5. Приложение не отправляло геолокацию на сервер

**Проблема:** Умная карта показывала только статичные демо-маркеры, реальная позиция пользователя не сохранялась и не отображалась у других.

**Решение:** В клиенте подключён `expo-location`: при переходе в режим «Умная карта» запрашивается разрешение, затем используется `watchPositionAsync` с интервалом обновления; текущие координаты и выбранный статус отправляются на сервер через `PUT /map/me`. Список маркеров других пользователей подгружается с `GET /map/users` и обновляется по таймеру.

### 6. Нужна регистрация пользователей

**Задача:** Дать пользователю возможность зарегистрироваться (email + имя), а не только пользоваться гостевым аккаунтом.

**Решение:** Добавлен экран «Профиль» (вкладка «Профиль» в табах) с отображением текущего пользователя. Если пользователь гость (`email` заканчивается на `@dogpaw.local`), показывается форма: Email, Имя, кнопка «Зарегистрироваться». Вызов `api.createUser({ email, name })` создаёт пользователя на сервере; ответ сохраняется в AsyncStorage и в контексте — текущий пользователь заменяется на зарегистрированного. Валидация: проверка формата email и непустых полей; при ошибке (например, email уже занят) показывается Alert.

### 7. Ручное добавление мест неудобно и быстро устаревает

**Проблема:** Для карты мест приходилось вручную заполнять точки в seed/БД. Это не масштабируется и быстро устаревает.

**Решение:** Интегрирован поиск через Яндекс Карты на бэкенде, добавлен эндпоинт `GET /api/v1/places/search` и кнопка обновления в UI. Теперь список мест можно подтягивать автоматически по региону карты и сохранять локально. Для работы нужен `YANDEX_MAPS_API_KEY`.

### 8. Нет сценария экстренного оповещения о пропаже питомца

**Проблема:** В ТЗ нужен функционал Lost&Found, но в приложении не было быстрого сценария для создания и отображения объявлений.

**Решение:** Введён полноценный режим «Потерялся питомец» с хранением в БД, API, кнопкой быстрого создания объявления из карточки питомца и отдельным режимом отображения на карте.

### 9. Нефункциональные требования по устойчивости были описаны, но не реализованы

**Проблема:** Ранее были только формулировки в документации; в коде отсутствовали меры защиты от перегрузки и от зависших сетевых запросов.

**Решение:** Добавлены ограничение частоты запросов на backend (rate limiting) и таймаут + retry на клиенте для API-запросов.

### 10. Большинство кнопок были заглушками

**Проблема:** Значительная часть UI была статичной: кнопки и списки визуально присутствовали, но не выполняли полезного действия.

**Решение:** Переведены основные пользовательские сценарии на реальные данные и API-взаимодействие: прогулки, лента, друзья, добавление друга по QR, статьи. Для кнопок, функционал которых отложен, добавлено явное действие (информирование пользователя), чтобы избежать «мертвых» нажатий.

### 11. Проверка работоспособности после рефакторинга экранов

**Проверено:**
- компиляция backend-пакетов: `go test ./... -run TestDoesNotExist -count=0` — успешно;
- smoke-тест API через локальные HTTP-запросы:
  - создание пользователей,
  - создание питомца,
  - добавление друга,
  - публикация поста,
  - создание прогулки,
  - создание объявления о пропаже и получение списка активных объявлений.

**Итог:** ключевые цепочки API работают, данные записываются и читаются из SQLite.

### 12. Проверка auth + realtime-привязки геолокации

**Проверено:**
- регистрация пользователя (`/auth/register`) и получение токена;
- чтение текущего профиля по токену (`/auth/me`);
- обновление своей позиции (`/map/me`) с Bearer-токеном;
- публикация поста (`/feed`) с Bearer-токеном.

**Итог:** `user_id` в геолокации и постах совпадает с аккаунтом из токена, что подтверждает корректную привязку данных к конкретному пользователю.

## Интересные решения

### Гостевой пользователь без регистрации

Чтобы не вводить полноценную аутентификацию на первом этапе, но при этом хранить данные по пользователям, реализована схема «гостевой пользователь»:

- При первом запуске вызывается `POST /users` с именем вида «Гость XXXXXX» и уникальным email — создаётся запись в БД.
- Возвращённые `id`, `name`, `email` сохраняются в AsyncStorage под ключом `@dogpaw_current_user`.
- При следующих запусках этот пользователь подставляется во все запросы (карта, питомец и т.д.). Таким образом, данные каждого пользователя однозначно привязаны к пользователю на сервере.

### Централизованный контекст пользователя

Вместо передачи `userId` через пропсы или глобальную переменную используется React-контекст `UserContext`:

- Обёртка `UserProvider` в `App.tsx` при монтировании инициализирует/создаёт текущего пользователя и кладёт его в контекст.
- Экраны (`MapScreen`, `PetProfileScreen` и др.) получают `user` через `useUser()` и используют `user.id` для запросов к API. Добавление новых экранов, которым нужен текущий пользователь, не требует проброса пропсов по всему дереву.

### Polling вместо WebSocket для умной карты

Обновление позиций других пользователей на карте сделано через периодический опрос `GET /map/users` (каждые 15 с), без WebSocket:

- Меньше изменений в бэкенде и проще развёртывание.
- Для прототипа и умеренной нагрузки достаточно; при росте числа пользователей можно позже заменить на WebSocket или Server-Sent Events.

### Разделение «Места» и «Умная карта» на одном экране

Один экран карты с переключателем режимов:

- В режиме «Места» загружаются только места (`GET /places`) и отображаются маркеры категорий (ветклиники, парки и т.д.).
- В режиме «Умная карта» дополнительно запрашивается геолокация, включается отправка своей позиции и подгрузка списка пользователей; маркеры других пользователей показываются с именем и статусом. Так избегаем лишних запросов и запросов геолокации, когда пользователь смотрит только места.

### Регистрация без отдельного экрана входа

Регистрация встроена в экран «Профиль» и не требует отдельного экрана входа на первом этапе:

- Пользователь по умолчанию — гость (создаётся при первом запуске). Он может пользоваться приложением и при желании перейти в «Профиль» и зарегистрироваться (email + имя).
- После регистрации гость в контексте и в AsyncStorage заменяется на нового пользователя; повторный вход (логин) можно добавить позже, используя тот же `POST /users` для проверки или отдельный эндпоинт входа. Так сохраняется простая модель «сначала гость, потом регистрация» без обязательного экрана логина до использования приложения.

### Гибридная модель карты мест: внешний поиск + локальная БД

Для быстрого UX и снижения числа внешних запросов выбрана гибридная схема:

- внешняя система (Яндекс) используется как источник актуальных POI;
- после получения точки сохраняются в локальную БД проекта;
- экран карты читает уже локальный `/places`, поэтому работает быстрее и стабильнее даже при временных сбоях внешнего API.

### Быстрый экстренный сценарий из карточки питомца

Чтобы уменьшить время реакции владельца в стрессовой ситуации, кнопка «Потерялся питомец» вынесена прямо в экран карточки:

- один тап создаёт объявление (с попыткой взять текущую геопозицию),
- объявление сразу появляется на карте в режиме Lost & Found.
